command vs args

Reference card · Pod container entrypoint & arguments

The model

Every image bakes in an ENTRYPOINT (the executable) and a CMD (default arguments). In a Pod spec:

  • command:  →  overrides ENTRYPOINT (the executable)
  • args:  →  overrides CMD (the arguments)

The container runs command + args concatenated. No shell is involved unless you explicitly add one (sh -c). A shell is just another program (/bin/sh) — it is not automatic.

When do you need sh -c?

Only when the command uses shell syntax:

FeatureExampleNeeds sh -c?
Plain binarysleep 3600No
Variable expansionecho $GREETINGYes
And / or / sequenceecho hi && sleep 3600Yes
Pipeps | grep nginxYes
Redirectdate >> /tmp/logYes
Globcat *.logYes

Just a binary + arguments → run it directly.
Anything with $, &&, |, >, ;, or * → wrap in sh -c.

The three forms

# 1. Direct binary — no shell (Exercise 1: sleeper)
command: ["sleep", "3600"]

# 2. Rely on image entrypoint, override only args
#    (what `kubectl run ... -- sleep 3600` generates)
args: ["sleep", "3600"]

# 3. Shell form — needed for $VAR, &&, pipes, redirects
#    (Exercise 3: envpod). Whole line is ONE string.
command: ["sh", "-c"]
args: ["echo $GREETING && sleep 3600"]

Literal vs expanded: command: ["echo", "$GREETING"] prints the literal text $GREETING — only a shell expands $.

One string, not words: in the sh -c form the entire shell line must be a single args string. If you split it into words, sh -c only runs the first word.

command vs args alone: args alone works only if the image has no ENTRYPOINT (true for busybox/alpine). command is the bulletproof form for any image — it replaces the executable outright.

Exam speed tip

# Generate scaffold, then hand-edit command/args/env/resources
k run sleeper --image=alpine:3.19 --dry-run=client -o yaml \
    -- sleep 3600 > sleeper.yaml

# Everything after `--` becomes the container args.
