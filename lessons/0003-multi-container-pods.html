Multi-Container Pods

Lesson 003 · Estimated time: 35 minutes · Domain: Application Design and Build (20%)

Real applications rarely run as a single lonely container. A web server needs its config generated first; a log shipper needs to tail the app's logs; a proxy needs to sit in front. The CKAD exam tests whether you can wire multiple containers into one Pod and make them cooperate. The two mechanisms — shared volumes and localhost networking — are the whole game.

The one idea: containers in a Pod cooperate two ways

Back in Lesson 002 you learned a Pod is "one or more containers that share a network namespace and storage." Now we use the "one or more" part. Containers in the same Pod cooperate through exactly two channels:

  • Shared network — they reach each other on localhost (same IP, different ports).
  • Shared storage — they mount the same volume and read/write the same files.

They do not share a filesystem by default. Each container has its own filesystem from its own image. Sharing happens only through a volume you explicitly mount into both.

The shared volume: emptyDir

The simplest shared volume is emptyDir — an empty directory created when the Pod starts, living as long as the Pod lives, and mountable into multiple containers. It is your default tool for container-to-container file sharing.[docs]

apiVersion: v1
kind: Pod
metadata:
  name: sharing
spec:
  volumes:                      # 1. declare the volume on the Pod
  - name: shared
    emptyDir: {}
  containers:
  - name: writer
    image: busybox:1.36
    command: ["sh", "-c", "echo hello > /data/msg.txt && sleep 3600"]
    volumeMounts:               # 2. mount it into each container
    - name: shared
      mountPath: /data
  - name: reader
    image: busybox:1.36
    command: ["sh", "-c", "sleep 5 && cat /data/msg.txt && sleep 3600"]
    volumeMounts:
    - name: shared
      mountPath: /data

spec.volumes[]
Volumes are declared once at the Pod level. emptyDir: {} means "give me a scratch directory."

spec.containers[].volumeMounts[]
Each container that wants the volume mounts it by name, choosing its own mountPath. The paths don't have to match — both point at the same underlying storage.

The name in volumeMounts must exactly match a name in spec.volumes. A typo here is the #1 reason a multi-container Pod silently fails to share. Always eyeball that the two names line up.

Init containers — run first, then get out of the way

Sometimes you need setup work done before the app starts: generate a config, wait for a dependency, clone a repo. That's an init container. It runs to completion before any app container starts, and they run sequentially in order.[docs]

apiVersion: v1
kind: Pod
metadata:
  name: init-demo
spec:
  volumes:
  - name: workdir
    emptyDir: {}
  initContainers:               # runs first, to completion
  - name: setup
    image: busybox:1.36
    command: ["sh", "-c", "echo 'config ready' > /work/config.txt"]
    volumeMounts:
    - name: workdir
      mountPath: /work
  containers:                   # starts only after initContainers succeed
  - name: app
    image: busybox:1.36
    command: ["sh", "-c", "cat /work/config.txt && sleep 3600"]
    volumeMounts:
    - name: workdir
      mountPath: /work

spec.initContainers[]
Same shape as a regular container. The difference is lifecycle: each one must exit 0 before the next runs, and all must finish before spec.containers start.

The classic handoff: the init container writes a file into a shared emptyDir, and the app container reads it. The volume outlives the init container, so the data is still there.

If an init container keeps running (e.g. you add sleep 3600), the app container never starts — init containers must finish. The pod sits in status Init:0/1. A long-running helper is a sidecar, not an init container (next section).

Sidecars — the long-running helper

A sidecar runs alongside the app for the whole Pod lifetime — a log shipper, a proxy, a metrics agent. There are two ways to express this, and you should know both:

1. The classic way — a second app container

Just add a second entry under spec.containers. Both run concurrently for the life of the Pod. This is what most exam questions mean by "sidecar," and it's the form you'll write most often:

spec:
  containers:
  - name: app          # main
    image: nginx:1.27
  - name: log-sidecar  # helper, runs concurrently
    image: busybox:1.36
    command: ["sh", "-c", "tail -F /var/log/app/access.log"]

2. The native way — an init container with restartPolicy: Always

Since Kubernetes v1.29 (stable), a "native sidecar" is an entry under initContainers that has restartPolicy: Always. This makes it start early (in init order), keep running, and shut down after the main container on termination.[docs]

spec:
  initContainers:
  - name: log-sidecar
    image: busybox:1.36
    restartPolicy: Always       # ← this turns an init container into a sidecar
    command: ["sh", "-c", "tail -F /var/log/app/access.log"]

If a question just says "add a sidecar that does X" with no ordering/lifecycle requirement, the classic second-container form is fastest and always correct. Reach for the native initContainers + restartPolicy: Always form when the question specifically needs the helper ready before the main app, or alive until after it stops.

Hands-on practice

Cluster running? k get nodes should show Ready. If not: kind create cluster --name ckad-practice.

Exercise 1 — shared volume. Write a Pod named combo with two busybox:1.36 containers sharing an emptyDir mounted at /data:

  • producer — writes the current date into /data/now.txt every 2 seconds, forever.
  • consumer — tails /data/now.txt.

Apply it, then verify the consumer sees the producer's writes:

k logs combo -c consumer
Solution 
cat <<EOF | k apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: combo
spec:
  volumes:
  - name: shared
    emptyDir: {}
  containers:
  - name: producer
    image: busybox:1.36
    command: ["sh", "-c", "while true; do date > /data/now.txt; sleep 2; done"]
    volumeMounts:
    - name: shared
      mountPath: /data
  - name: consumer
    image: busybox:1.36
    command: ["sh", "-c", "tail -F /data/now.txt"]
    volumeMounts:
    - name: shared
      mountPath: /data
EOF

Note k logs needs -c <container> when a Pod has more than one container — otherwise it asks you which one.

Exercise 2 — init container handoff. Write a Pod named initapp where:

  • an init container seed (busybox:1.36) writes "seeded by init" into /work/data.txt;
  • the main container app (busybox:1.36) prints that file and then sleeps 3600.

Verify the handoff worked:

k logs initapp           # should print: seeded by init
k get pod initapp        # READY 1/1, STATUS Running
Hint

Both the init container and the app container must mount the same volume name. The init container's command must exit — no sleep in it.

Solution 
cat <<EOF | k apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: initapp
spec:
  volumes:
  - name: workdir
    emptyDir: {}
  initContainers:
  - name: seed
    image: busybox:1.36
    command: ["sh", "-c", "echo 'seeded by init' > /work/data.txt"]
    volumeMounts:
    - name: workdir
      mountPath: /work
  containers:
  - name: app
    image: busybox:1.36
    command: ["sh", "-c", "cat /work/data.txt && sleep 3600"]
    volumeMounts:
    - name: workdir
      mountPath: /work
EOF

Exercise 3 — diagnose the trap (no writing). Apply this Pod, then explain what you observe and why:

cat <<EOF | k apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: stuck
spec:
  initContainers:
  - name: never-ends
    image: busybox:1.36
    command: ["sh", "-c", "sleep 3600"]
  containers:
  - name: app
    image: busybox:1.36
    command: ["sh", "-c", "sleep 3600"]
EOF
k get pod stuck

What STATUS does stuck show, and why won't app start? How would you fix it if never-ends was supposed to keep running? (Answer to your agent — this one's a discussion.) Clean up with k delete pod stuck combo initapp when done.

You can now build multi-container Pods three ways: a shared emptyDir for file handoff, an initContainer for sequential setup, and a sidecar (classic or native) for a long-running helper. You know the cooperation channels are localhost (network) and shared volumes (storage), and that volume name must match between volumes and volumeMounts.

Key mental models

  • No sharing by default. Containers in a Pod share storage only through a volume mounted into both; they share network via localhost.
  • Init = finishes; sidecar = keeps running. If a helper must exit before the app, it's an init container. If it lives alongside, it's a sidecar.
  • The volume outlives the init container. That's how data written by init survives for the app to read.
  • Name matching is everything. volumeMounts[].name must equal volumes[].name.
