Pods from First Principles

Lesson 002 · Estimated time: 30 minutes · Domain: Application Design and Build (20%)

Every workload in Kubernetes runs inside a Pod. Deployments, Jobs, DaemonSets — they all create Pods. If you understand the Pod spec, you understand the building block that everything else wraps. The exam will test whether you can write and modify Pod manifests quickly and correctly.

What is a Pod, really?

A Pod is one or more containers that share a network namespace and storage. That's the core idea.

What "share a network namespace" means concretely:

  • All containers in a Pod share the same IP address
  • Containers in the same Pod can reach each other via localhost
  • Port conflicts happen within a Pod (two containers can't both use port 80)

Think of a Pod as a tiny machine. The containers inside it are processes on that machine.

The Pod manifest — anatomy

When you ran k run nginx --image=nginx, Kubernetes generated a full Pod spec. Let's look at what a minimal Pod manifest looks like when you write it yourself:

apiVersion: v1
kind: Pod
metadata:
  name: nginx
  labels:
    app: nginx
spec:
  containers:
  - name: nginx
    image: nginx:1.27
    ports:
    - containerPort: 80

Every field explained

apiVersion: v1
Pods are part of the core API group, so it's just v1. No group prefix needed.

kind: Pod
The resource type. Case-sensitive — always capital P.

metadata.name
Unique identifier within the namespace. Must be a valid DNS subdomain (lowercase, hyphens OK, no underscores).

metadata.labels
Key-value pairs used for selection. Services and Deployments find Pods by label. Not required for bare Pods, but always good practice.

spec.containers[]
Array of containers. At minimum: name and image. The name is how you reference the container in logs, exec, etc.

spec.containers[].ports[].containerPort
Informational — declares what port the container listens on. Doesn't actually open anything (unlike Docker's -p). Useful for documentation and for Services to know where to send traffic.

The imperative shortcut — and its limits

You already know k run nginx --image=nginx creates a Pod. But the exam often requires modifications that imperative commands can't express. The power move is:

# Generate the YAML without creating the pod
k run nginx --image=nginx --dry-run=client -o yaml > pod.yaml

# Edit it to add what you need
vim pod.yaml

# Apply it
k apply -f pod.yaml

--dry-run=client -o yaml is your best friend in the exam. It generates correct boilerplate instantly. You then tweak only what the question asks for. This is faster than writing YAML from scratch, and less error-prone.

Common Pod spec fields (exam-relevant)

Beyond the minimum, here are the fields you'll need most often:

apiVersion: v1
kind: Pod
metadata:
  name: app
  labels:
    app: myapp
spec:
  restartPolicy: Always          # Always | OnFailure | Never
  containers:
  - name: app
    image: busybox:1.36
    command: ["sh", "-c"]        # Override entrypoint
    args: ["echo hello && sleep 3600"]  # Override CMD
    env:
    - name: MY_VAR
      value: "hello"
    resources:
      requests:
        memory: "64Mi"
        cpu: "250m"
      limits:
        memory: "128Mi"
        cpu: "500m"
    ports:
    - containerPort: 8080

restartPolicy
Always (default) — restarts container if it exits for any reason. OnFailure — only restarts on non-zero exit. Never — no restart. Jobs use OnFailure or Never.

command
Overrides the container image's ENTRYPOINT. Note: it's an array, not a string.

args
Overrides the container image's CMD. Also an array.

env
Environment variables. Can also pull from ConfigMaps and Secrets (covered in a later lesson).

resources.requests
What the scheduler uses to place the Pod. "I need at least this much."

resources.limits
Hard ceiling. Container gets OOMKilled if it exceeds memory limit.

Hands-on practice

Make sure your cluster is running (kind create cluster --name ckad-practice if needed).

Exercise 1: Generate a Pod manifest for an alpine:3.19 container named sleeper that runs the command sleep 3600. Don't create it yet — just generate the YAML.

Hint 
k run sleeper --image=alpine:3.19 --dry-run=client -o yaml -- sleep 3600
Solution 
k run sleeper --image=alpine:3.19 --dry-run=client -o yaml -- sleep 3600 > sleeper.yaml
cat sleeper.yaml

The -- separates kubectl flags from the container command. Everything after -- becomes the container's args.

Exercise 2: Write a Pod manifest from scratch (no --dry-run) for a pod named web running nginx:1.27 with these requirements:

  • Label: tier: frontend
  • Resource request: 64Mi memory, 100m CPU
  • Resource limit: 128Mi memory, 200m CPU
  • Container port: 80

Apply it and verify it's running.

Solution 
cat <<EOF | k apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: web
  labels:
    tier: frontend
spec:
  containers:
  - name: web
    image: nginx:1.27
    ports:
    - containerPort: 80
    resources:
      requests:
        memory: "64Mi"
        cpu: "100m"
      limits:
        memory: "128Mi"
        cpu: "200m"
EOF

Verify: k get pod web — should show STATUS: Running within a few seconds.

Exercise 3: Create a pod named envpod using busybox:1.36 that:

  • Sets an environment variable GREETING=hello-ckad
  • Runs the command: echo $GREETING && sleep 3600

After creating it, verify the env var works by running:

k exec envpod -- env | grep GREETING
Solution 
cat <<EOF | k apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: envpod
spec:
  containers:
  - name: envpod
    image: busybox:1.36
    command: ["sh", "-c"]
    args: ["echo $GREETING && sleep 3600"]
    env:
    - name: GREETING
      value: "hello-ckad"
EOF

Then: k exec envpod -- env | grep GREETING should output GREETING=hello-ckad

If you completed all three exercises, you can now write Pod manifests from memory. You understand the core fields: metadata, spec.containers, command/args, env, resources, and ports.

Key mental models

  • Pod = shared network + shared storage. Containers in a Pod are like processes on one machine.
  • Imperative for speed, declarative for correctness. Use k run ... --dry-run=client -o yaml to generate, then edit.
  • command overrides ENTRYPOINT, args overrides CMD. If you set command, the image's CMD is ignored entirely unless you also set args.
  • Requests = scheduling, Limits = enforcement. A Pod with requests but no limits can use more than it asked for (until the node runs low).
